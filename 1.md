## Bloco 1: A Anatomia da Web

A internet não é uma nuvem abstrata, mas sim uma rede física de computadores que se comunicam através de protocolos estritos.

### 1.1 Arquitetura Cliente-Servidor
Toda interação na Web segue este modelo:
*   **Cliente:** A entidade que *inicia* a comunicação pedindo um recurso. Pode ser um navegador web, um aplicativo mobile ou um script Python rodando no terminal.
*   **Servidor:** Um computador (ou cluster) que fica *escutando* a rede, recebe os pedidos, processa (muitas vezes consultando um banco de dados) e devolve uma resposta.

### 1.2 Domínio, DNS e Hospedagem
Para acessar um servidor na internet, precisamos saber seu endereço. Computadores usam endereços IP numéricos (ex: `192.0.2.1`), mas humanos preferem nomes.

*   **Domínio:** O nome amigável que digitamos no navegador (ex: `viacep.com.br`, `github.com`).
*   **DNS (Domain Name System):** O serviço que traduz domínios em endereços IP. Quando você acessa `viacep.com.br`, seu computador consulta um servidor DNS para descobrir qual IP corresponde a esse nome.
*   **Hospedagem (Hosting):** O serviço que mantém os arquivos e o código do servidor rodando em um computador conectado à internet 24 horas por dia. Quando contratamos uma hospedagem, estamos alugando espaço e capacidade de processamento em um servidor remoto.

**Fluxo simplificado de uma requisição:**
```
1. Usuário digita: https://viacep.com.br/ws/01001000/json/
2. Navegador consulta DNS: "Qual IP corresponde a viacep.com.br?"
3. DNS responde: "165.227.126.241"
4. Navegador conecta-se ao servidor em 165.227.126.241 e envia a requisição HTTP
5. Servidor processa a requisição e devolve a resposta (JSON com o endereço)
6. Navegador (ou script Python) recebe e interpreta a resposta
```

### 1.3 Requisição (Request) e Resposta (Response)
Uma comunicação HTTP é dividida em duas partes:

**A Requisição (O que o cliente envia):**
1.  **Verbo (Método):** Indica a *intenção* da ação.
    *   `GET`: Solicitar dados (ex: buscar um CEP).
    *   `POST`: Enviar dados para criar um novo recurso (ex: cadastrar um usuário).
    *   `PUT` / `PATCH`: Atualizar um recurso existente.
    *   `DELETE`: Remover um recurso.
2.  **Headers (Cabeçalhos):** Metadados sobre a requisição. Exemplos: `User-Agent` (quem é o cliente), `Accept` (formato esperado, como `application/json`), `Authorization` (tokens de acesso).
3.  **Body (Corpo):** Os dados em si. Obrigatório em verbos como `POST` e `PUT`, geralmente formatados em **JSON**.

**A Resposta (O que o servidor devolve):**
1.  **Código de Status:** Um número de 3 dígitos indicando o resultado.
2.  **Headers:** Metadados da resposta (ex: `Content-Type`, `Set-Cookie`).
3.  **Body:** O dado solicitado (ex: o JSON com o endereço) ou uma mensagem de erro.

### 1.4 Códigos de Status HTTP
Aprender a ler estes códigos é essencial para depuração:
*   **2xx (Sucesso):**
    *   `200 OK`: A requisição funcionou e o corpo contém o resultado.
    *   `201 Created`: Um novo recurso foi criado com sucesso (comum em `POST`).
*   **3xx (Redirecionamento):**
    *   `301 Moved Permanently`: O recurso mudou de URL definitivamente.
*   **4xx (Erro do Cliente):** O cliente fez algo errado.
    *   `400 Bad Request`: A requisição está malformada.
    *   `401 Unauthorized`: Falta autenticação.
    *   `403 Forbidden`: Autenticado, mas sem permissão para acessar o recurso.
    *   `404 Not Found`: O recurso (URL) não existe no servidor.
*   **5xx (Erro do Servidor):** O servidor falhou ao processar um pedido válido.
    *   `500 Internal Server Error`: Erro genérico no código do servidor (ex: banco de dados caiu).
    *   `502 Bad Gateway`: O servidor atua como proxy e não conseguiu resposta do servidor principal.

📚 **Referência:** [MDN Web Docs - HTTP Status Codes](https://developer.mozilla.org/pt-BR/docs/Web/HTTP/Status)

## Bloco 2: Tratamento de Erros em Python (`try`/`except`)

Antes de consumir APIs, precisamos aprender a lidar com situações onde algo pode dar errado. Em Python, usamos a estrutura `try`/`except` para isso.

### 2.1 Conceito Básico
Quando executamos código que depende de fatores externos (como uma conexão de rede), o programa pode falhar. Em vez de deixar o programa travar com uma mensagem de erro críptica, podemos "capturar" o erro e tratar a situação de forma controlada.

**Sintaxe:**
```python
try:
    # Código que pode falhar
    resultado = alguma_funcao_que_pode_erro()
except TipoDoErro:
    # Código que roda SE o erro ocorrer
    print("Ocorreu um erro, mas o programa não travou.")
```

### 2.2 Exemplo Prático
```python
def dividir(a, b):
    try:
        return a / b
    except ZeroDivisionError:
        return "Não é possível dividir por zero."
    except TypeError:
        return "Ambos os argumentos devem ser números."

print(dividir(10, 2))   # 5.0
print(dividir(10, 0))   # "Não é possível dividir por zero."
print(dividir(10, "x")) # "Ambos os argumentos devem ser números."
```

### 2.3 Múltiplos `except` e `Exception` genérico
Podemos capturar diferentes tipos de erro em blocos separados, ou usar um `except Exception:` genérico para capturar qualquer erro (use com cautela, pois pode esconder bugs).

```python
try:
    # código arriscado
    pass
except ConnectionError:
    print("Erro de conexão com a rede.")
except TimeoutError:
    print("A operação demorou muito.")
except Exception as e:
    print(f"Ocorreu um erro inesperado: {e}")
```

📚 **Referência:** [Python Tutorial - Errors and Exceptions](https://docs.python.org/3/tutorial/errors.html)

### 3.2 Consumindo a API do ViaCEP
O [ViaCEP](https://viacep.com.br/) é um webservice gratuito para consultar CEPs nacionais.

```python
import requests

# 1. Fazendo a requisição GET
url = "https://viacep.com.br/ws/01001000/json/"

try:
    resposta = requests.get(url, timeout=5)  # timeout evita espera infinita
    resposta.raise_for_status()  # Lança exceção se status for 4xx ou 5xx
    
    # 2. Convertendo o Body de JSON (texto) para Dicionário Python
    dados = resposta.json()
    
    # 3. Validando a regra de negócio (ViaCEP retorna {"erro": true} para CEP inválido)
    if "erro" in dados:
        print("CEP não encontrado.")
    else:
        print(f"Logradouro: {dados.get('logradouro')}")
        print(f"Bairro: {dados.get('bairro')}")
        print(f"Cidade/UF: {dados.get('localidade')}/{dados.get('uf')}")
        
except requests.exceptions.ConnectionError:
    print("Erro: Não foi possível conectar ao servidor. Verifique sua internet.")
except requests.exceptions.Timeout:
    print("Erro: A requisição demorou muito e foi cancelada.")
except requests.exceptions.HTTPError as e:
    print(f"Erro HTTP: {e}")
except Exception as e:
    print(f"Erro inesperado: {e}")
```

### 3.3 A Pegadinha da Validação
O que acontece se consultarmos um CEP inexistente, como `99999999`?

```python
resposta_invalida = requests.get("https://viacep.com.br/ws/99999999/json/")
print(resposta_invalida.status_code) # Imprime: 200
print(resposta_invalida.json())      # Imprime: {'erro': True}
```

**Lição:** O Status Code `200 OK` significa apenas que *o servidor HTTP respondeu com sucesso à requisição*. Não significa que a *regra de negócio* foi um sucesso. O ViaCEP optou por retornar 200 e incluir uma chave `"erro": true` no JSON. Sempre valide a estrutura e o conteúdo do JSON retornado, não apenas o status HTTP.

📚 **Referência:** [Requests Quickstart](https://requests.readthedocs.io/en/latest/user/quickstart/)
